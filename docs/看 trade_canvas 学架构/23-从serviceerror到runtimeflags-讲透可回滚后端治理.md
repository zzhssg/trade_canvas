# 第23关：从 ServiceError 到 RuntimeFlags，讲透可回滚后端治理

上一关你学了"依赖怎么管、边界怎么守"。这一关解决一个更现实的问题：

**系统能不能在"持续迭代 + 持续上线"下，依然可控、可回滚、可排障？**

想象一家医院。门诊不会让所有病人直接冲进手术室——先挂号分诊，再按科室处理，遇到危重情况有紧急预案，每一步都有记录可追溯。

后端系统也一样。真正拖垮项目的往往不是"算法写错"，而是：

- 错误边界混乱（所有病人都挤急诊，轻症重症分不清）；
- 配置来源混乱（每个科室自己定规矩，互相矛盾）；
- 紧急回滚手段缺失（出了事只能拔电源）；
- 关停阶段不可观测（医院关门时病人还在走廊游荡）。

trade_canvas 把这套治理做成了四层保险。

---

## 0. 先给一句总纲

可回滚架构 = 错误分层 + 配置单真源 + kill-switch + 关停可观测

1. **错误分层**：业务层不直接绑框架异常。（分诊台）
2. **配置单真源**：运行期开关只能从统一入口注入。（中央药房）
3. **kill-switch**：高风险能力默认关，必要时一键熔断。（紧急停堆按钮）
4. **关停可观测**：停服务时也尽量"有秩序"，减少噪音与误报。（有序疏散预案）

这四层合起来，才叫"可回滚"而不是"碰运气"。

---

## 1. 第一层保险：错误分层——先把"业务错误"和"HTTP 传输"拆开

很多项目会在 service 里直接 `raise HTTPException`。
短期快，长期会出现两个问题：

- 业务层被 FastAPI 绑死，单测要跟着框架跑。
- 同一类业务错误在不同 route 里表达不一致，前端难处理。

这就像医院里护士直接给病人开药方——跳过了医生诊断环节，出了问题谁也说不清。

trade_canvas 现在改成：

- service/read-model 抛 `ServiceError`（医生写诊断书）
- route 层统一 `to_http_exception()` 映射（前台翻译成病人能懂的话）

这等于把"业务语义"和"传输协议"做了边界隔离。

你只需要记住一句：

**service 负责"错了什么"，route 负责"怎么回给 HTTP"。**

---

## 2. 第二层保险：配置单真源——避免运行期散读 env

另一个常见坑：
代码到处 `os.environ`，今天改一处生效、明天另一处忘改，行为漂移非常难查。

这就像医院里每个科室自己定用药标准——心内科说阿司匹林 100mg，骨科说 200mg，同一个病人在不同科室得到不同治疗。

现在的治理口径是：

- `RuntimeFlags` 统一读取 env（中央药房统一配药）
- `container` 启动时注入（药房把药送到各科室）
- runtime/service 只消费注入结果，不再临时读环境变量（科室不许自己去药房拿药）

这件事看起来"只是整洁"，实际收益非常大：

1. 行为可复现：同一组 flags 就是同一行为。
2. 测试可控：单测不需要到处 monkeypatch env。
3. 变更可审计：开关都在同一个结构体里看得见。

---

## 3. 第三层保险：kill-switch 先行，尤其是补偿类逻辑

最容易"救火变事故"的地方，是补偿逻辑。
例如 ingest 主链里，overlay 失败后要不要补偿重置？新写入 candle 要不要回滚？

这里采用了标准策略：

- 能改变主链路语义的能力都要有 `TRADE_CANVAS_ENABLE_*`
- 默认关闭
- 逐步放开

你可以把这理解成核电站的紧急停堆按钮：

- 功能异常时，不必立刻回滚代码（不必拆反应堆）；
- 先关开关，恢复主链稳定（先停堆，控制住局面）；
- 再离线修复与复盘（再排查具体原因）。

这就是工程里常说的"把风险隔离在可操作面"。

---

## 4. 第四层保险：服务关停也要有秩序

很多人只关注"运行时成功路径"，忽略"关停路径"。
结果是 E2E 虽然通过，但日志全是 `CancelledError`，难以分辨真假问题。

这就像医院关门时不做疏散——病人还在走廊游荡，护士不知道该不该继续输液，监控室全是红灯但没人知道哪个是真警报。

本项目做了两层收口：

1. **应用层**：在 shutdown 阶段对被取消的 HTTP 请求做受控响应（503），减少无意义异常栈。
2. **脚本层**：E2E cleanup 改成"先 TERM + 等待，再必要时 KILL"，减少残留进程与收尾抖动。

注意这不是"掩盖问题"，而是把"预期中的取消"从"异常噪音"变成"可解释行为"。

---

## 5. 这套治理为什么对初学者也重要

你可能会想：我先把算法写对不就行了？

答案是：**算法正确只是第一步，可控迭代才是长期能力。**

当你进入多人协作和持续交付后，真正拖垮项目的往往不是"公式写错"，而是错误边界混乱、配置来源混乱、紧急回滚手段缺失、关停阶段不可观测。

这关学会后，你会从"能写功能"进化为"能守住系统"。

---

## 6. 这套治理背后的通用方法论

你可以把可回滚治理抽象成一套通用模板：

1. **错误先分层**：业务语义和传输协议必须隔离，service 管"错了什么"，route 管"怎么回"。
2. **配置单真源**：所有运行参数从一个入口注入，禁止散读环境变量。
3. **高风险默认关**：能改变主链路语义的能力必须有 kill-switch，且默认关闭。
4. **关停要有序**：shutdown 阶段的取消不是异常，而是可解释行为。

四层合起来的核心收益：**线上出问题时，你的第一反应不是"改代码重新部署"，而是"关一个开关"。**
